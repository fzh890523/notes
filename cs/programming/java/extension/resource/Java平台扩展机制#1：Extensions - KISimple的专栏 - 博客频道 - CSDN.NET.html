    Java平台扩展机制#1：Extensions

    标签： javaextension
    2014-07-29 23:12 541人阅读 评论(0) 收藏 举报
    本文章已收录于：
    分类：

    版权声明：本文为博主原创文章，未经博主允许不得转载。

    概述

    The extension mechanism provides a standard, scalable way to make custom APIs available to all applications running on the Java platform.


    举个栗子，假设你将Google的Guava库作为Extension来扩展Java平台，那么你便可以像使用Java Collections Framework一样使用Google Guava Collections，看上去GGC就好像变成了Java平台的一部分。

    Extension以jar包的形式存在，下面就来看看如何将一个jar包变成Extension。

    two ways

    Java平台提供了两种方式的Extension

    1. Installed Extensions，基于约定。如下图，只要是在特定jre底下的lib/ext里的jar包都会作为平台的Extension，这种形式有个明显的缺陷，多个jdk无法共享相同的jar包作为Extension，到了Java6增加了个系统属性 java.ext.dirs 来指定ext目录，这样只要放到这个目录下，多个jdk也可以共享这个Extension了。
    2. Downloaded Extensions，基于配置。只要在平台jar包（不管是自带的还是扩展的）的manifest配置上另一个jar包，被配置的这个jar包就会被当作是Extension（有点类似maven的依赖机制）。举个栗子，假设你在rt.jar的manifest文件配置了GGC，那么GGC就会被当作Extension，与放在/lib/ext的效果是一样的。配置的方式有两种，Class-Path和Extension-List，具体戳这里。


    Classloading Order

    现在来考虑两个问题：

    1. 当有一个类，即存在于平台的core classes（rt.jar等）又存在于Extension，名字完全相同，当你要去加载这个类的时候，会使用哪一个？
    2. 如果上述的这种冲突是存在于两个Extension那情况又是如何？

    第一个问题，Java平台的解决方案是，给出了类加载顺序：

    1. Bootstrap classes. rt.jar属于这一趴。
    2. Installed extensions.
    3. The class path. CLASSPATH或者-cp指定。Downloaded Extension应该是属于这一趴。

    第二个问题，似乎应该是依赖于具体JVM实现的，后面再讨论哈。

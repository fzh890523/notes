原文：Zero Copy I: User-Mode Perspective


几乎所有人都听说过Linux中所谓的零拷贝功能，但我经常遇到一些并没有完全理解它的人。因此，我决定写一些文章深入阐述这个主题，希望可以将这个有用的特性解释清楚。在这篇文章中，我们将从用户态应用程序的视角来一窥零拷贝，因此会有意隐藏掉复杂的内核实现细节。

什么是零拷贝？

为了更好的理解一个问题的解决方案，我们需要先理解这个问题本身。我们先来看一下一个文件存储服务器为客户端提供服务的简单过程。请看示例代码：

[cpp] view plain copy
 print?
  1. read(file, tmp_buf, len);  
  2. write(socket, tmp_buf, len);  
read(file, tmp_buf, len);
write(socket, tmp_buf, len);

看上去很简单；你可能会觉得只有这两个系统调用并不会有太多开销。事实上，相去甚远。在这两个调用背后，数据至少被拷贝了四次，而且用户态/内核态的上下文切换也是如此。（事实上这个过程要复杂得多，但我想简单点说）。为了更好的理解这个过程，请看下图。图中上半部展示了上下文切换的过程，下半部展示了拷贝的过程。

  1. read调用会产生一次用户态到内核态的上下文切换。DMA引擎会执行第一次拷贝，文件内容会从磁盘读取到内核空间的缓冲区。
  2. 数据会从内核缓冲区拷贝到用户缓冲区，read调用返回。这个返回会产生一次内核态到用户态的上下文切换。现在数据已经在用户缓冲区，程序可以继续了。
  3. write调用会产生一次用户态到内核态的上下文切换。第三次的拷贝产生，数据被拷贝到内核空间的缓冲区。这个缓冲区是与socket相关联的。
  4. write调用返回，产生第四次上下文切换。

第四次的拷贝是DMA引擎将数据从内核缓冲区发送给协议引擎，独立并且异步完成。你可能会问，"独立并且异步是几个意思？难道数据不是在调用返回之前就已经被传输过去了吗？"。事实上，调用返回并不能保证数据传输完成；甚至不能保证数据已经开始传输。它只能说明在以太网驱动的队列里有空闲的描述符，并且驱动接收了数据传输的请求。可能会有大量的数据包排在我们的数据前面。除非驱动实现了优先环/*priority rings*/或者优先队列，否则数据传输就是遵循先进先出的原则。（上图中叉装的DMA copy说明最后一次DMA拷贝有可能会延迟）

就像你看到的，非常多的数据拷贝其实都是没有必要的。为了减少开销提高性能，有一些拷贝是应该去掉的。作为一个驱动开发者，我会接触到拥有高级特性的硬件。一些硬件完全可以绕开内存，直接将数据传输给其他设备。这个特性可以消除一份系统内存中的拷贝，这很好，但并不是所有硬件都支持。这里还有另一个问题，数据要直接从磁盘传输到网络，是需要重新包装/*repackage*/的，这会带来一些复杂性。为了减少开销，我们可以从消除内核态与用户态间的拷贝入手。

一种方式是，不使用read调用，使用mmap。举个栗子：

[cpp] view plain copy
 print?
  1. tmp_buf = mmap(file, len);  
  2. write(socket, tmp_buf, len);  
tmp_buf = mmap(file, len);
write(socket, tmp_buf, len);

为了更好的理解这个过程，请看下图。上下文的切换还是一样。

  1. mmap调用会使得DMA引擎将文件内容拷贝到内核缓冲区。这个缓冲区是与用户进程共享的，不会产生内核与用户内存空间的拷贝。
  2. write调用会将数据从原来的内核缓冲区拷贝到与socket关联的内核缓冲区。
  3. DMA引擎将数据从socket缓冲区发送给协议引擎，产生第三次拷贝。

通过使用mmap而不是read，我们将内核需要的拷贝减半了。当有大量数据要传输时，这会有很好的效果。然后，这是需要付出代价的。使用mmap+write的方式是有一些隐藏的缺陷的。其中有一个坑就是，当你把一个文件映射到内存并执行write调用时，有其他的进程对那个文件执行了截断操作。你的write调用会被总线错误信号SIGBUS中断，因为你访问了错误的内存地址。对该信号的默认处理就是结束进程并执行coredump，这并不是一个网络服务器最想要的。有两种方法应对这个问题。

第一种方式是注册一个SIGBUS的信号处理器，简单的执行返回。如果这么做，write调用就会返回中断前成功写出去的字节数，errno会被设为成功。这种治标不治本的方式可能会产生糟糕的情况。因为产生SIGBUS意味着进程发生了严重的错误，我并不鼓励使用这种方案。

第二种方法要使用内核的文件租赁/*file leasing*/（Windows中叫做机会锁/*opportunistic locking*/）。这是解决这个问题的正确方法。通过租赁文件描述符，你跟内核就订立了一份该文件的租约。你可以向内核请求读写该租约。当其他进程想要截断你正在传输的文件时，内核会给你发送一个实时信号RT_SIGNAL_LEASE 。它告诉你内核将要毁约了。在进程访问到一个无效地址并被结束前，write调用就会被中断。返回值是中断前写出去的字节数，errno会被设置为成功。下面的示例代码展示了如何跟内核订立一份租约：

[cpp] view plain copy
 print?
  1. if(fcntl(fd, F_SETSIG, RT_SIGNAL_LEASE) == -1) {  
  2.     perror("kernel lease set signal");  
  3.     return -1;  
  4. }  
  5. /* l_type can be F_RDLCK F_WRLCK */  
  6. if(fcntl(fd, F_SETLEASE, l_type)){  
  7.     perror("kernel lease set type");  
  8.     return -1;  
  9. }  
if(fcntl(fd, F_SETSIG, RT_SIGNAL_LEASE) == -1) {
    perror("kernel lease set signal");
    return -1;
}
/* l_type can be F_RDLCK F_WRLCK */
if(fcntl(fd, F_SETLEASE, l_type)){
    perror("kernel lease set type");
    return -1;
}

使用mmap之前需要先拿到租约，操作结束之后再跳出租约，通过调用fcntl(fd, F_SETLEASE, F_UNLCK)实现。

sendfile

在内核2.1版本，为了简化网络数据传输还有本地文件传输引入了sendfile系统调用。sendfile不仅减少了数据拷贝，也减少了上下文切换。这样使用它：

[cpp] view plain copy
 print?
  1. sendfile(socket, file, len);  
sendfile(socket, file, len);

为了更好的理解这个过程，请看下图。

  1. sendfile调用会使得DMA引擎将文件内容拷贝到内核缓冲区。然后数据再被拷贝到内核socket缓冲区。
  2. DMA引擎将数据从内核socket缓冲区发送给协议引擎，产生第三次拷贝。

你可能会问，当我们调用sendfile传输文件时，另一个进程截断了这个文件会怎么样？如果我们没有注册任何的信号处理器，sendfile会简单的返回它被中断之前写出去的字节数，并将errno设置为成功。

如果我们在调用sendfile之前与内核订下租约，效果也是一样的。在sendfile返回之前我们也会收到RT_SIGNAL_LEASE信号。

到目前为止我们已经可以避免内核的多次拷贝，但仍然还是存在一份拷贝。可以把它也去掉吗？当然，但需要借助一点硬件的帮助。要消除所有的内核拷贝，我们需要一个支持聚合操作的网络接口。这意味着等待传输的数据不需要在连续的内存当中；它可以被分散在内存的不同位置。在内核2.4版本中，对socket缓冲区描述符做了修改以适应那样的需求—就是大家所知道的Linux零拷贝。这种方法不仅减少了上下文切换，也消除了处理器所做的数据拷贝。对用户级程序来说没有什么不同，所以代码看起来还是像这样：

[cpp] view plain copy
 print?
  1. sendfile(socket, file, len);  
sendfile(socket, file, len);

为了更好的理解这个过程，请看下图。

  1. sendfile调用会使得DMA引擎将文件内容拷贝到内核缓冲区。
  2. 数据不会被拷贝到socket缓冲区。只有关于位置信息和数据大小的描述符会被追加到socket缓冲区中。DMA引擎直接将数据从内核缓冲区发送给协议引擎，因此最后一份拷贝也没掉了。

实际上数据还是要从磁盘拷贝到内存，再从内存拷贝到网络，所以可能有人会说这并不是真正的零拷贝。从操作系统的立场来看，这确实是零拷贝，因为数据在内核缓冲区中只有一份。使用零拷贝，除了避免数据冗余还会有许多其他的性能提升，例如更少的上下文切换，更少的CPU数据缓存污染，还避免了CPU校验和计算。
